## 1. 문서 목적

이 문서는 라벨 자동생성 프로그램의 요구사항, 자동 바인딩 규칙, UX 개선 방향, 그리고 현재 프로젝트에 실제로 구현되어 있는 기능을 한 문서로 통합 정리한 최종 기준 문서다.

정리 원칙은 다음과 같다.

1. 원문 요청(`라벨_자동생성_요청원문.md`)의 업무 요구를 기준으로 삼는다.
2. `v0.2`, `v1.0`, `v1.1` 문서에 흩어진 규칙과 방향성을 흡수한다.
3. 최종 판단은 현재 코드가 실제로 수행하는 동작 기준으로 정리한다.
4. 기존 문서 중 현재 구현과 불일치하는 내용은 "현재 구현 기준"으로 바로잡아 기록한다.

참고:
- `라벨_자동바인딩_명세_v0.2.md`는 현재 안내문만 남아 있고, 실제 바인딩 규칙은 `v1.0`, `v1.1`, 요청원문, 현재 코드에 흡수되어 있다.

## 2. 프로젝트 개요

### 2.1 목표

엑셀 주문 데이터를 업로드한 뒤, 식품 라벨에 필요한 핵심 정보만 추출하고, 관리 마스터 데이터를 이용해 자동 바인딩된 라벨을 생성한 후, 사용자가 필요한 필드만 수정하여 즉시 인쇄할 수 있는 오프라인 데스크톱 앱을 제공한다.

### 2.2 현재 범위

- 오프라인 Electron 데스크톱 앱
- 서버/DB 없음
- 로컬 JSON 저장
- Windows 설치형 실행 파일 배포
- macOS 기능 테스트 가능
- iOS 앱 범위 제외

### 2.3 핵심 사용자 흐름

1. 엑셀 파일 업로드
2. 핵심 컬럼만 추출하여 작업표 생성
3. 관리 데이터(포장단위, 품명 사전 등) 등록 또는 수정
4. 선택한 행 기준으로 라벨 자동 생성
5. 라벨별 경고 확인 및 필드 수정
6. 인쇄 전 검수 모달 확인
7. 실제 인쇄 실행

## 3. 현재 기술 구성

### 3.1 런타임 및 패키지

- Electron `28.3.3`
- `xlsx` `0.18.5`
- `electron-builder` `24.13.3`
- `playwright`는 UI 캡처/검수용 개발 의존성으로 존재

### 3.2 주요 파일 역할

| 파일 | 역할 |
|---|---|
| `main.js` | Electron 메인 프로세스, 창 생성, 저장소 파일 읽기/쓰기 IPC |
| `preload.js` | 렌더러에 `desktopApi` 브리지 노출 |
| `renderer/index.html` | 전체 단일 페이지 UI 구조 |
| `renderer/app.js` | 상태 관리, 파싱, 마스터 CRUD, 자동 바인딩, 라벨 생성/편집/인쇄 로직 |
| `renderer/styles.css` | 레이아웃, 편집 UI, 인쇄 미디어 스타일 |
| `package.json` | 실행/빌드 스크립트 및 Windows 빌드 설정 |

```bash

```

#### Windows 배포 빌드


산출물:

## 5.5 자동 바인딩 규칙

## 5.5.1 공통 규칙

- 라벨 생성 단위: 엑셀 1행 -> 분할 수만큼 라벨
- 선택된 추출 행만 생성
- 동일 원본 행에서 기존 라벨이 있으면 새 생성 전 교체

### 라벨 타입 판정

- 어류 키워드 포함 시 `어류`
- 또는 품명 사전에서 `type=어류` 키워드 매칭 시 `어류`
- 그 외는 `육류`

### 원산지 판정

현재 코드 상수 키워드:

- 국내산
- 중국산
- 러시아산
- 미국산
- 호주산
- 캐나다산
- 뉴질랜드산
- 베트남산
- 칠레산
- 노르웨이산

### 용도 판정

1. 용도 마스터 값이 상품명에 포함되면 우선 사용
2. 없으면 `OO용` 패턴 정규식 탐지

### 규격 판정

1. 규격 마스터 일치 우선
2. 없으면 `n*n*n` 패턴 탐지

## 5.5.2 포장 단위 및 중량 분할

### 육류

- 포장 단위는 `packUnits` 마스터에서 탐색
- 탐색 순서
  1. 정규화된 식당명 exact match
  2. 정규화된 식당명 partial match
- 빈 지점명 행은 partial match에서 제외

### 어류

- 포장 단위를 고정 `10kg`로 사용

### 중량 분할 계산

- `fullCount = floor(total / packUnit)`
- `remain = total - fullCount * packUnit`
- `remain > 0`이면 마지막 라벨 1장 추가
- 포장 단위가 없거나 0 이하이면 전체 중량 1장으로 처리

## 5.5.3 육류 바인딩

### 제조일자

- 기본: 납품일자 하루 전
- 단, 납품일이 월요일이면 3일 전(금요일)

### 구분

- 상품명에 `냉동` 포함 -> `냉동`
- 그 외 -> `냉장`

### 보관방법

- 냉동 -> `-18'C 이하 냉동보관`
- 냉장 -> `-2'C ~ -5'C 냉장보관`

### 소비기한

- 냉장 + 일반포장 -> 제조일 + 14일
- 냉장 + 진공포장 -> 제조일 + 45일
- 냉동 -> 제조일 + 365일

### 품목보고번호 / 원재료

- 품명 기준 `productReports` exact match

### 이력번호 / 도축장

- 품명 기준 `traceability` exact match 목록 조회
- 우선순위
  1. `availableKg >= requiredKg` 충족 여부
  2. `slaughterDate` 최신순

## 5.5.4 어류 바인딩

### 품명

- 품명 사전에서 `type=어류` 기준 매칭
- 매칭 실패 시 빈값으로 두고 경고 처리

### 제조년월일

- 납품일이 속한 주의 월요일을 구한 뒤
- 그 월요일 기준 직전 금요일

### 납품일 표시

- `YYYY-MM-DD (요일)` 형식 사용

### 소비기한

- 제조일 + 365일

### 기본값

- 품목제조보고번호 미매핑 시 `2025027955015`
- 원재료 미매핑 시 `{품명 또는 어류} 100%`
- 고정 안내문구는 어류 전용 상수 사용

## 5.5.5 공통 고정값

### 제조원/판매원/식품유형/허가번호

- 기본값 프로필에서 주입

### 고정 안내문구

- 육류: `MEAT_NOTICE`
- 어류: `FISH_NOTICE`

### 현재 경고 종류

## 5.8 라벨 편집

### 현재 편집 방식

- 좌측에 생성 라벨 그룹 목록 표시
- 우측에 선택 라벨 편집 폼 표시
- 동일한 필드/경고 구성을 가진 라벨은 그룹으로 묶음

### 목록 표시 정보

- 라벨 구분
- 장수 배지
- 품명
- 납품처
- 중량 표시

- 경고 요약

### 편집 폼 섹션

1. 라벨 기본
2. 유통/보관
3. 원료/이력
4. 사업자 정보
5. 기타

### 편집 가능한 필드

- 사실상 모든 라벨 필드 직접 수정 가능
- 텍스트, 셀렉트, 텍스트에어리어 혼합

### 수동 수정 정책

- 사용자가 수정한 필드는 `manualOverrides[field] = true`
- UI에 `수동수정` 배지 표시
- 자동 바인딩은 최초 생성 시 1회 수행

### 소비기한 재계산

아래 필드가 바뀌면 소비기한을 다시 계산한다.

- `labelType`
- `manufactureDate`
- `storageType`
- `packagingType`

### 경고 패널 기능

- 현재 라벨 경고 수 표시
- 경고별 해결 가이드 표시
- 관련 필드로 이동 버튼
- 다음 경고 라벨 이동 버튼
- 경고가 없을 때도 다른 경고 라벨로 이동 가능

### 현재 한계

- 다중 라벨 일괄 편집 없음
- 선택 필드만 재바인딩 기능 없음

## 11. 결론

현재 프로젝트는 원문 요청의 핵심 업무인 "엑셀 업로드 -> 핵심 데이터 추출 -> 마스터 기준 자동 바인딩 -> 라벨 편집 -> 인쇄" 흐름을 실제로 수행할 수 있는 상태다.

기존 `v1.0`, `v1.1` 문서에서 계획으로만 남아 있던 여러 UX 개선 항목도 현재 코드에 상당 부분 반영되었다. 반면, 권한 분리, 상세 변경 이력, 재바인딩 모드 세분화, 파싱 에러 리포트 같은 운영 고도화 기능은 아직 후속 과제로 남아 있다.
